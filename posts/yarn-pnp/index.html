Yarn PnP 적용기 — node_modules를 버리고 얻은 것들

node_modules의 문제

파일이 너무 많다

nodemodules 안에 nodemodules 안에 node_modules. 중첩 구조가 끝없이 이어진다. 디렉토리 구조가 복잡해지고, 파일 수가 기하급수적으로 늘어난다.

NPM은 이 문제를 호이스팅으로 해결하려 했다. 모든 모듈을 최상단 node_modules로 끌어올리는 방식이다. 하지만 이건 또 다른 문제를 만들었다.

유령 의존성(Phantom Dependencies)

package.json에 명시하지 않은 패키지를 사용할 수 있는 상황이 생긴다. 호이스팅 때문에 다른 패키지의 의존성이 최상단에 올라오면서, 직접 설치하지 않았는데도 require가 된다.

이건 혼란을 야기한다. 어느 날 갑자기 의존성 트리가 바뀌면 찾을 수 없다는 에러가 터진다.

느린 모듈 탐색

NPM은 패키지를 찾기 위해 계속 상위 디렉토리의 node_modules 폴더를 탐색한다. 바로 찾지 못할수록 readdir, stat 같은 느린 I/O 호출이 반복된다. 경우에 따라서는 I/O 호출이 중간에 실패하기도 한다.

프로젝트 내부뿐 아니라 개발자 로컬 글로벌 영역까지 검색한다. 환경에 따라 결과가 달라질 수 있다.

안 되면 node_modules 지우고 재설치가 국룰인 건 다 이유가 있다.

Plug’n’Play (PnP) 방식

Yarn의 접근은 단순하다. node_modules를 아예 만들지 않는다.

기존 node_modules는 파일 I/O를 발생시키며 파일시스템에 의존해서 모듈을 찾는다. PnP는 모듈의 위치를 명시적으로 작성해놓고 찾는다.

.yarn/cache 폴더에 의존성이 Zip 아카이브로 저장되고, .pnp.cjs 파일에 의존성을 찾을 수 있는 정보가 기록된다.

// .pnp.cjs에 명시된 정보에 따라 각 패키지들이 동적으로 참조된다
const lodashCeilPath = require.resolve('lodash/ceil');
// → /path/to/.yarn/cache/lodash-npm-4.17.11-xxxxx.zip/node_modules/lodash/ceil.js

PnP 런타임이 Node의 fs 모듈을 패치해서 Zip 아카이브 내부 파일에 직접 접근할 수 있게 해준다. 별도 작업 없이 기존 코드가 그대로 동작한다.

PnP의 이점

  • 디스크 용량 감소: node_modules보다 훨씬 적은 용량
  • 탐색 속도 향상: 폴더를 순회할 필요 없이 .pnp.cjs에서 바로 위치를 찾음
  • 유령 의존성 해결: 호이스팅을 하지 않으므로 명시하지 않은 패키지는 접근 불가

실제 비교: node_modules vs PnP

동일한 프로젝트 기준으로 비교했다.

node_modules PnP
설치 시간 73.78s 6s
디스크 용량 1.2 GB 189 MB

넘사벽이다.

설정 방법

.yarnrc.yml에서 linker 모드를 설정한다.

# PnP 모드 (기본값)
nodeLinker: pnp

# 기존 node_modules 방식으로 되돌리려면
nodeLinker: node-modules

Zero Install

PnP의 진짜 킬러 피처는 Zero Install이다. .yarn/cache에 있는 Zip 파일들을 Git에 커밋해두면, clone 후 yarn install 없이 바로 실행할 수 있다.

.gitignore 설정:

# Zero Install을 사용할 때
.yarn/*
!.yarn/cache
!.yarn/patches
!.yarn/plugins
!.yarn/releases
!.yarn/sdks
!.yarn/versions

# Zero Install을 사용하지 않을 때
.yarn/*
!.yarn/patches
!.yarn/releases
!.yarn/plugins
!.yarn/sdks
!.yarn/versions
.pnp.*

Zero Install을 쓰면 CI에서 install 단계가 사라진다. 새 팀원이 clone하고 바로 yarn start를 칠 수 있다. 의존성 설치 관련 이슈가 원천 차단된다.

마무리

node_modules는 Node.js 생태계의 오래된 관습이다. 동작은 하지만 구조적으로 느리고, 유령 의존성 같은 예측 불가능한 문제를 안고 있다.

Yarn PnP는 이 문제를 근본적으로 해결한다. 설치 시간 12배 단축, 디스크 용량 84% 절감. 숫자가 모든 걸 말해준다.

물론 PnP와 호환되지 않는 패키지가 간혹 있어서, packageExtensions나 patchedDependencies 설정이 필요할 수 있다. 하지만 대부분의 메이저 라이브러리는 이미 PnP를 지원한다. 아직 node_modules를 쓰고 있다면, 전환을 고려해볼 만하다.
