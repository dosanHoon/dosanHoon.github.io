Claude Code 커스텀 커맨드로 프론트엔드 워크플로우 자동화하기

왜 커스텀 커맨드인가

Claude Code를 쓰다 보면 반복되는 패턴이 생긴다. 브랜치 따고 리베이스하고, 변경사항 커밋하고 PR 올리고, 배포하고 테스트 돌리고. 매번 같은 순서의 명령을 Agent에게 자연어로 설명하는 건 비효율적이다.

Claude Code는 프로젝트 루트의 .claude/commands/ 폴더에 마크다운 파일을 넣으면, 그걸 슬래시 커맨드(/)로 실행할 수 있다. 한번 정의해두면 팀원 누구나 같은 워크플로우를 한 줄로 실행할 수 있다.

나는 이걸 활용해서 프론트엔드 개발의 반복 작업을 거의 전부 커맨드로 만들었다. 현재 운영 중인 커맨드는 크게 5개 카테고리, 12개다.

커맨드 구조

.claude/commands/
├── worktree.md        # 브랜치 체크아웃 + 리베이스 + 워크트리
├── pr.md              # 커밋 → 푸시 → PR 생성
├── review-pr.md       # PR 코드 리뷰
├── dev-deploy.md      # DEV 서버 배포
├── dev-test.md        # 유닛 + E2E 테스트
├── dev-deploy-and-test.md  # 배포 + 테스트 코드 작성 + 실행
├── add-eventlog.md    # 이벤트 로그 누락 분석 + 코드 추가
├── eventlog-deploy.md # 이벤트 로그 추가 → 배포 → 일감 등록
├── qa-fetch.md        # QA 일감 수집
├── qa-plan.md         # QA 이슈 분석 + Fix Plan
├── qa-fix.md          # QA 코드 수정 + 테스트 + 커밋
└── qa-auto-fix.md     # QA 전체 파이프라인 자동화

각 .md 파일 안에 Agent가 따라야 할 절차를 마크다운으로 작성한다. Claude Code는 이걸 읽고 그대로 실행한다.

Git 워크플로우 자동화

가장 자주 쓰는 건 /worktree와 /pr이다.

/worktree — 브랜치 생성부터 워크트리까지 한 줄

/worktree feature/upload-watermark

이 한 줄이 실행하는 것:

  1. feature/upload-watermark 브랜치가 없으면 생성
  2. develop 기준으로 리베이스
  3. Git worktree로 별도 디렉토리에 체크아웃

왜 worktree를 쓰냐면, 모노레포에서 브랜치를 전환하면 node_modules 충돌이나 빌드 캐시 오염이 생기기 때문이다. Worktree로 분리하면 메인 작업 디렉토리를 건드리지 않고 병렬로 작업할 수 있다.

/pr — 변경사항을 PR까지 한번에

/pr

이것만 치면:

  1. 변경사항 분석 후 적절한 커밋 메시지 생성
  2. 리모트에 푸시
  3. PR 제목과 본문 자동 작성
  4. GitHub PR 생성

커밋 메시지나 PR 본문을 매번 고민할 필요가 없다. Agent가 diff를 분석해서 작성하고, 마음에 안 들면 수정하면 된다.

배포 + 테스트 자동화

/dev-deploy-and-test — 가장 강력한 커맨드

/dev-deploy-and-test

이 커맨드는 세 단계를 연결한다:

  1. 현재 브랜치를 DEV 서버에 배포
  2. 변경된 코드에 대한 테스트가 없으면 테스트 코드 작성
  3. Vitest(유닛) + Playwright(E2E) 테스트 실행

“테스트 코드가 없으면 작성”하는 부분이 핵심이다. 기능 구현은 빠르게 하고 싶은데 테스트 작성은 미루게 되는 경향이 있다. 이 커맨드를 쓰면 배포할 때 자동으로 테스트가 따라온다. Agent가 변경된 컴포넌트를 분석해서 적절한 테스트 케이스를 생성한다.

QA 파이프라인 자동화

QA 단계에서 가장 시간을 많이 잡아먹는 건 이슈 확인 → 원인 파악 → 수정 → 재배포 사이클이다. 이걸 4단계 커맨드로 분리했다.

/qa-fetch    → QA 시스템에서 이슈 수집
/qa-plan     → 이슈 분석 + 수정 계획 수립
/qa-fix      → 코드 수정 + 테스트 + 커밋
/qa-deploy   → DEV 서버에 배포

각 단계를 독립적으로 실행할 수 있어서, 계획만 먼저 확인하고 싶을 때는 /qa-plan까지만, 한번에 끝내고 싶을 때는 /qa-auto-fix로 전체 파이프라인을 돌린다.

/qa-auto-fix

이 한 줄이 fetch → plan → fix → deploy를 순차 실행한다. 단순한 UI 버그는 사람이 개입하지 않아도 수정되는 경우가 많다.

이벤트 로그 자동화

프론트엔드에서 빠뜨리기 쉬운 게 이벤트 로그다. 기능은 구현했는데 분석용 로그를 안 심는 경우가 많다.

/add-eventlog feature/upload-watermark

이 커맨드는:

  1. 해당 브랜치의 변경사항을 분석
  2. 새로 추가된 사용자 인터랙션(버튼 클릭, 페이지 진입 등) 중 이벤트 로그가 누락된 부분을 찾아냄
  3. 로그 코드를 자동 추가

분석팀에서 “이 버튼 클릭 이벤트가 안 잡히는데요”라고 물어오는 상황을 사전에 방지할 수 있다.

커맨드 작성 팁

실제로 운영하면서 배운 몇 가지.

절차를 명확하게 쓴다. Agent가 판단할 여지를 최소화해야 한다. “적절히 커밋해”가 아니라 “변경 파일을 분석해서 conventional commit 형식으로 커밋 메시지를 생성하고, 파일 단위로 스테이징해서 커밋한다”처럼 구체적으로 쓴다.

실패 케이스를 명시한다. “테스트가 실패하면 수정하고 재실행한다. 3회 이상 실패하면 실패 로그를 출력하고 중단한다” 같은 fallback을 적어둬야 Agent가 무한 루프에 빠지지 않는다.

파이프라인은 단계별로 분리한다. /qa-auto-fix처럼 전체를 한번에 돌리는 커맨드도 만들되, 각 단계(fetch, plan, fix, deploy)를 개별 커맨드로도 실행할 수 있게 한다. 디버깅할 때 단계별로 확인할 수 있어야 한다.

MCP 서버와 조합한다. 커맨드 안에서 Jira MCP로 일감을 가져오거나, Figma MCP로 기획서를 참조하게 할 수 있다. 커맨드 단독보다 MCP와 조합했을 때 자동화 범위가 크게 넓어진다.

결과

이 커맨드 체계를 운영한 뒤 체감되는 변화는 명확하다.

PR 생성에 걸리는 시간이 줄었다. 커밋 메시지 고민, PR 본문 작성, 리뷰어 지정 같은 부수 작업을 Agent가 처리한다. QA 이슈 대응도 빨라졌다. /qa-auto-fix 한 줄로 단순 이슈는 사람이 코드를 열어보지 않아도 수정된다.

무엇보다 팀 전체가 같은 워크플로우를 쓰게 된다는 점이 크다. .claude/commands/는 Git으로 관리되니까, 한 사람이 커맨드를 개선하면 팀 전체에 적용된다. “배포는 어떻게 하더라?”같은 질문이 사라진다.

시작하기

커스텀 커맨드를 처음 만든다면, 가장 자주 반복하는 작업 하나를 골라서 시작하는 걸 추천한다. 브랜치 생성이든, PR 올리기든, 배포든. 그 작업을 할 때 터미널에 치는 명령어들을 순서대로 마크다운에 적으면 그게 첫 번째 커맨드가 된다.
