AI Agent 시대, 마크다운 문서 체계가 팀 생산성을 바꾼다

들어가며

AI Agent(Claude, Cursor, Copilot 등)를 활용해 코딩하는 팀이 점점 늘고 있습니다. 그런데 Agent에게 기능 구현을 시키려면, 결국 프로젝트의 맥락을 전달해야 합니다. “이 기능의 스펙이 뭔지”, “지난 회의에서 어떻게 결정했는지” 같은 것들이요.

저는 최근 사내 프로젝트에서 기획서, 회의록, PRD 등을 전부 마크다운으로 관리하는 체계를 구축했습니다. 단순한 문서 관리가 아니라, AI Agent가 직접 참조할 수 있도록 설계한 것입니다. 이 글에서는 왜 이런 방식을 선택했고, 어떤 효과가 있었는지 공유합니다.

왜 마크다운인가?

프로젝트 문서는 보통 Confluence, Notion, Google Docs 같은 도구에 작성합니다. 사람이 읽기엔 좋지만, AI Agent에게 전달하기엔 몇 가지 문제가 있습니다.

첫째, API를 통해 가져와야 하므로 매번 네트워크 호출이 필요합니다. 둘째, HTML이나 JSON 형태로 반환되어 파싱 과정에서 토큰이 낭비됩니다. 셋째, 권한 설정과 인증 과정이 복잡합니다.

마크다운은 이런 문제를 깔끔하게 해결합니다. 파일 시스템에 바로 존재하므로 Agent가 직접 읽을 수 있고, 텍스트 기반이라 토큰 효율이 높으며, Git으로 버전 관리가 자연스럽습니다.

문서 구조 설계

실제로 제가 구성한 문서 레포의 구조는 이렇습니다.

project-docs/
├── CLAUDE.md              # Agent를 위한 프로젝트 가이드
├── README.md              # 프로젝트 개요
├── specs/                 # 기능 명세서
│   ├── common.md
│   ├── search.md
│   ├── upload.md
│   ├── settings.md
│   └── admin.md
├── meetings/              # 회의록
│   ├── milestone-1.md
│   ├── milestone-2.md
│   └── external.md
└── planning/              # 기획 자료
    ├── prd/               # Product Requirements Document
    ├── flows/             # 시스템/사용자 플로우 (Mermaid)
    └── interviews/        # 사용자 인터뷰

핵심은 기능 단위로 문서를 나누는 것입니다. 화면 단위가 아닌 기능 단위로 작성해야 Agent가 특정 기능을 구현할 때 필요한 문서만 정확히 참조할 수 있습니다.

Agent별 진입점 파일 — 동일한 원칙, 각자의 규격

AI Agent마다 프로젝트에 진입할 때 읽는 설정 파일이 다릅니다. 저는 세 가지 주요 Agent에 맞춰 각각의 파일을 만들되, 동일한 내용을 넣어두었습니다.

project-docs/
├── CLAUDE.md                          # Claude Code용
├── .cursorrules                       # Cursor용
└── .github/copilot-instructions.md    # GitHub Copilot용

세 파일 모두 같은 내용을 담고 있습니다. 문서 구조 설명, specs 파일 목록, meetings 파일 목록, 문서 업데이트 방법 등이요. 이렇게 하면 팀원이 어떤 Agent를 쓰든 동일한 프로젝트 맥락을 전달받게 됩니다.

# 프로젝트 문서 가이드

이 레포는 서비스의 기능 명세서와 회의록을 관리합니다.
코드 구현 시 관련 명세서를 반드시 참고하세요.

## 문서 구조

- `specs/` — 기획서에서 추출한 기능 명세서 (기능 단위)
- `meetings/` — 마일스톤별 회의록 요약

## specs 파일 목록

| 파일 | 내용 |
|------|------|
| `search.md` | 목록 조회, 상세보기, 필터, 다운로드 |
| `upload.md` | 파일 업로드, 요청 프로세스 |
| ...  | ... |

이 파일들이 있으면 Agent는 “다운로드 기능을 수정해줘”라는 요청을 받았을 때, 자신의 규격에 맞는 가이드 파일을 읽고 → specs/search.md에서 다운로드 관련 명세를 찾아 → 정확한 구현을 할 수 있습니다. Agent에게 매번 장황하게 설명할 필요가 없어지는 것입니다.

한 가지 팁이라면, 내용을 하나의 원본에서 관리하고 나머지는 복사하는 방식을 추천합니다. 세 파일의 내용이 달라지면 Agent별로 다른 맥락을 전달받게 되니까요.

문서 업데이트 자동화

문서의 원본은 여전히 Figma(기획서)와 Confluence(회의록)에 있을 수 있습니다. 마크다운 문서는 이들의 미러링본이며, Agent를 통해 동기화할 수 있습니다.

Figma에서 기획서 동기화

Figma MCP 서버를 활용하면, Agent가 직접 Figma 파일에 접근해 기획 내용을 마크다운으로 변환할 수 있습니다.

사용자: "검색 기능 기획서 최신화해줘"
Agent: [Figma MCP로 해당 페이지 조회]
     → [텍스트 추출 및 구조화]
     → [specs/search.md 업데이트]

Confluence에서 회의록 동기화

Confluence REST API를 사용하면 회의록도 자동으로 가져올 수 있습니다. 한 가지 팁이라면, Confluence API는 V2를 사용하는 게 좋고 Basic Auth 방식이 안정적입니다. 이런 삽질의 결과도 CLAUDE.md에 기록해두면, 다음에 Agent가 같은 작업을 할 때 같은 실수를 반복하지 않습니다.

기억 의존 제거

팀 개발에서 가장 큰 문제 중 하나는 구두 합의입니다. 회의에서 “A 방식으로 가자”고 결정했는데, 일주일 뒤에 “그때 어떻게 하기로 했지?”라고 물어보는 상황은 누구나 겪어봤을 것입니다.

회의록을 마크다운으로 축적하면 이런 상황을 완전히 제거할 수 있습니다.

## 2026-01-15 워터마크 정책 논의

### 결정사항
- 워터마크는 다운로드 시점에 서버에서 적용
- 미리보기에는 클라이언트 측 워터마크 사용
- 워터마크 해제는 2차 인증 후 가능

### 배경
- 기존에는 원본 파일이 그대로 노출되는 이슈가 있었음
- 보안팀 요청으로 워터마크 정책 강화

이렇게 쌓인 회의록 덕분에 “워터마크 정책이 어떻게 되었지?”라고 물으면, Agent가 관련 회의록을 검색해서 바로 답을 줍니다. 사람의 기억력에 의존하던 부분이 사라지는 것입니다.

Context와 Token 절약

AI Agent에게 프로젝트 맥락을 전달하려면 엄청난 양의 토큰이 필요합니다. 하지만 잘 정리된 마크다운 문서가 있으면, 필요한 부분만 선택적으로 참조할 수 있습니다.

❌ 비효율적인 방법
"우리 프로젝트는 이런 서비스인데, 조회 기능이 있고,
필터링이 있고, 다운로드가 있고, 권한 체계는 이러저러하고..."
→ 매번 긴 프롬프트로 설명 (수천 토큰 소비)

✅ 효율적인 방법
"specs/search.md를 참고해서 다운로드 기능 수정해줘"
→ Agent가 필요한 파일만 읽음 (수백 토큰으로 정확한 컨텍스트)

더 나아가, 축적된 문서를 주기적으로 Agent에게 요약시켜서 최신 상태의 요약본을 유지할 수도 있습니다. 대부분의 질문에는 요약본으로 충분하고, 상세 내용이 필요할 때만 원본 문서를 참조하면 됩니다.

팀 정보 정렬 (Team Alignment)

이 문서 체계의 가장 큰 가치는 팀원 간 정보 비대칭 해소입니다.

신규 팀원이 합류했을 때, “프로젝트 docs 레포를 읽어봐”라고 하면 됩니다. Agent가 문서를 기반으로 질문에 답해줄 수 있으니, 온보딩 비용이 크게 줄어듭니다.

기획자가 Figma에서 기획서를 업데이트하면 → Agent가 마크다운으로 동기화하고 → 개발자가 구현할 때 최신 기획을 자동으로 참조하는 흐름이 만들어집니다. 기획 변경 사항을 슬랙에서 일일이 공유하거나, “이거 반영된 거 맞아?” 하고 확인하는 커뮤니케이션 비용이 크게 줄어듭니다.

실전 활용 패턴

패턴 1: 기능 구현 시 명세서 참조

사용자: "업로드에 요청 프로세스 추가해줘"
Agent: [CLAUDE.md 읽기]
     → [specs/upload.md에서 요청 프로세스 명세 확인]
     → [meetings/milestone-2.md에서 관련 결정사항 확인]
     → [명세에 맞는 구현]

패턴 2: 과거 결정사항 조회

사용자: "외부 접근 정책 어떻게 결정됐지?"
Agent: [meetings/ 폴더 검색]
     → [관련 회의록에서 결정사항 추출]
     → [정리된 답변 제공]

패턴 3: 문서 최신화

사용자: "이번 주 회의 내용 문서에 반영해줘"
Agent: [Confluence API로 최신 회의록 조회]
     → [해당 마일스톤 회의록 파일에 추가]
     → [관련 specs 파일도 업데이트]
     → [Git 커밋]

도입 시 고려사항

이 방식을 도입할 때 몇 가지 배운 점이 있습니다.

문서 레포는 코드 레포와 분리하세요. 문서 변경과 코드 변경의 주기가 다르고, 접근 권한도 다를 수 있습니다. 기획자나 PM도 문서 레포에는 접근할 수 있어야 하니까요.

파일명은 직관적으로. search.md, upload.md처럼 영문 키워드를 사용하되, 필요하다면 한글 설명을 병기하면 Agent와 사람 모두에게 친화적입니다.

Agent별 가이드 파일은 규격에 맞게. Claude Code는 CLAUDE.md, Cursor는 .cursorrules, GitHub Copilot은 .github/copilot-instructions.md를 읽습니다. 세 파일에 동일한 내용을 넣어두면 어떤 Agent를 쓰든 같은 맥락을 전달받습니다.

삽질 기록도 문서화. API 연동 시 겪은 이슈, 버전별 차이점 같은 것들을 CLAUDE.md에 기록해두면 Agent가 같은 실수를 반복하지 않습니다.

마치며

마크다운 문서 체계는 단순한 문서 관리 방법이 아니라, AI Agent 시대의 팀 협업 인프라라고 생각합니다. 기획서와 회의록을 마크다운으로 축적하면 기억에 의존하지 않고, 토큰을 절약하며, 팀원 간 정보를 정렬할 수 있습니다.

실제로 이 체계가 자리 잡은 뒤로는, Agent에게 기능 구현을 시킬 때 “이 파일 참고해”라는 한마디면 충분해졌습니다. 매번 프로젝트 맥락을 장황하게 설명하던 시절과 비교하면 생산성 차이가 확실합니다. AI Agent를 적극 활용하는 팀이라면 한번 도입해볼 만한 방식입니다.
