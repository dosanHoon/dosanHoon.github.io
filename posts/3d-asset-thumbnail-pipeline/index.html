6만 건 3D 에셋에 썸네일을 입히기까지

문제: 3D 모션 에셋의 썸네일이 필요

사내 게임 에셋 라이브러리를 운영하고 있었다. 에셋은 BVH, FBX, GLB 세 가지 포맷이 혼재된 3D 데이터다.

썸네일이 하나도 없었다. 에셋 리스트 UI와 에셋 상세에 노출 하기 위해서 썸네일이 필요하다.

6만건의 에셋. 건당 2분만 잡아도 6만 건이면 2,000시간. 자동화가 필수였다.

방향: 블렌더 실패, 웹 플레이어로 전환

처음에는 블렌더(Blender) 스크립트로 썸네일을 뽑으려는 시도가 있었다. 팀장님이 진행했는데, 6만 건의 포맷이 제각각이라 블렌더 하나로 일괄 처리하기가 어려웠다. BVH/FBX/GLB 포맷별로 로딩 방식이 다르고, 렌더링 설정도 달라야 했다.

이 시점에서 내가 진행하기로 방향이 전환됐다. 우리에게는 이미 프로덕션에서 돌아가는 Three.js 기반 모션 플레이어가 있었다. 세 가지 포맷을 모두 지원하고, 캐릭터 렌더링과 모션 재생이 검증된 코드였다. 이 플레이어를 Puppeteer(Headless Chrome)로 감싸서 자동화하는 방향으로 잡았다.

별도의 서버사이드 렌더링 엔진을 구축하는 대신, 기존 코드를 재사용한 것이 핵심 판단이었다.

GIF에서 WebM으로 — 중간에 포맷을 바꾼 이유

프로젝트 초기에는 GIF로 썸네일을 만들었다. gif.js로 Three.js 캔버스를 프레임별로 캡처해서 GIF 애니메이션을 생성하는 방식이었다.

배치를 돌려보니 세 가지 문제가 드러났다.

용량이 컸다. GIF는 256색 제한인데도 파일 하나가 수MB에 달했다. 6만 건이면 스토리지 비용도 문제지만, 클라이언트에서 이걸 로딩하는 시간이 더 문제였다.

품질이 나빴다. 3D 렌더링 특성상 조명과 그림자의 그라데이션이 많은데, 256색으로 디더링되면 눈에 띄게 열화된다. 썸네일로서 에셋의 형태를 파악하는 데 지장이 있는 수준이었다.

브라우저 렌더링 성능이 나빴다. 에셋 목록에서 GIF 수십 개를 동시에 재생하면 브라우저가 버벅였다.

배치 생성 시간, 결과물 품질, 실제 서비스에서의 렌더링 성능 — 세 가지를 모두 고려해야 했다. GIF는 셋 다 부족했다.

대안을 찾다가 WebP(정지 이미지) + WebM(VP9 프리뷰 영상) 조합으로 전환했다. 결과는 확연했다. 용량은 GIF 대비 10~50배 줄었고, VP9 인코딩 덕분에 화질도 유지됐다.

프로젝트 중반에 포맷을 바꾸는 건 부담이었지만, 초기 선택에 매몰되지 않은 게 결과적으로 좋은 판단이었다.

에셋 목록 UI: video 태그 수십 개를 어떻게 처리할 것인가

포맷을 바꾸는 것만으로는 끝이 아니었다. 실제 서비스에서 에셋 목록을 보여주는 UI가 문제였다.

에셋 리스트에 WebM video 태그를 수십 개 동시에 만들어서 재생시킬 수는 없다. video 태그 하나하나가 디코더 리소스를 점유하고, 동시 재생 수가 늘어나면 브라우저 탭이 느려지거나 영상이 로딩되지 않는다.

해결 전략은 이렇다.

기본 상태에서는 WebP 정지 이미지만 노출한다. 목록을 스크롤할 때는 정지 썸네일만으로 에셋의 형태를 파악할 수 있다. 이미지이므로 수십 개를 동시에 렌더링해도 부담이 없다.

마우스 호버 또는 viewport 진입 시 WebM video로 교체한다. 사용자가 특정 에셋에 관심을 보일 때만 영상을 로딩하고 재생한다. 동시 재생되는 video 태그 수를 최소화하는 것이 핵심이다.

이 방식으로 “목록에서 빠르게 훑어보기”와 “개별 에셋의 모션 확인하기”를 분리했다.

배치 자동화: 6만 건을 어떻게 끝냈나

렌더링 파이프라인 자체는 Puppeteer로 Headless Chrome을 띄우고, 기존 모션 플레이어(CaptureApp)를 로딩해서 프레임을 캡처한 뒤, FFmpeg으로 VP9 WebM을 인코딩하는 구조다.

6만 건을 실제로 돌리면서 부딪힌 문제는 주로 안정성이었다.

메모리 누수. Three.js 리소스(지오메트리, 텍스처, 스켈레톤)가 에셋을 처리할 때마다 누적되어 Headless Chrome이 뻗었다. 20개 에셋마다 브라우저 탭을 닫고 새로 열어 메모리를 확보하는 방식(Page Recycling)으로 해결했다.

병렬 처리. 한 프로세스로 순차 처리하면 며칠이 걸렸다. 워커를 N개 띄우되, 별도의 큐 서버 없이 Stride(modulo) 방식으로 에셋을 분배했다. i번째 에셋은 i % N번 워커가 처리하는 단순한 구조다. 워커가 중간에 죽어도 같은 파라미터로 재시작하면 되고, 이미 완료된 건은 건너뛰는 옵션(SKIP_EXISTING)으로 이어하기가 가능했다.

포맷별 예외 처리. BVH, FBX, GLB 세 포맷이 각각 로딩 방식이 다르고, 같은 GLB라도 자체 모델이 포함된 파일과 모션만 담긴 파일이 섞여 있었다. 포맷별 분기 처리가 상당히 까다로웠는데, 기존 모션 플레이어가 이미 이 세 포맷을 지원하고 있었기 때문에 CaptureApp으로 추출하면서 그대로 활용할 수 있었다.

최종적으로 약 30시간에 6만 건 전체 처리를 완료했다.

결과

에셋 라이브러리에 썸네일이 적용된 뒤, 아티스트가 에셋을 파일명으로 추측하거나 하나씩 열어보는 일이 사라졌다. 목록에서 WebP 썸네일로 형태를 파악하고, 마우스를 올리면 최대 12초(15fps)의 모션 프리뷰를 바로 확인할 수 있게 됐다.

산출물 용도
WebP 정지 썸네일 (목록 기본 노출)
WebM (VP9) 모션 프리뷰 영상 (호버 시 재생)

회고

기존 코드를 재사용하라. 블렌더가 실패한 뒤 웹 플레이어로 방향을 전환한 것이 이 프로젝트의 가장 중요한 판단이었다. 이미 프로덕션에서 검증된 Three.js 렌더링 코드를 Puppeteer로 감싸는 것만으로 배치 도구가 됐다. 3D 렌더링 파이프라인을 새로 구축했다면 몇 배의 시간이 들었을 것이다.

초기 결정에 매몰되지 마라. GIF → WebM 전환은 프로젝트 중반에 이루어졌다. 이미 GIF 기반으로 배치를 돌리고 있는 상태에서 포맷을 바꾸는 건 심리적으로 쉽지 않았지만, 용량·품질·성능 세 가지 모두에서 도약이었다.

브라우저 렌더링 비용을 먼저 생각하라. 썸네일을 “만드는” 것만큼 중요한 게 “보여주는” 것이다. 6만 건의 프리뷰 영상을 동시에 재생할 수 없다는 제약을 인식하고, WebP 기본 노출 + 호버 시 WebM 교체라는 전략을 일찍 잡은 게 서비스 품질에 직접적으로 영향을 줬다.

영역 기술
3D 렌더링 Three.js
브라우저 자동화 Puppeteer (Headless Chrome)
영상 인코딩 FFmpeg (libvpx-vp9)
프레임 포맷 WebP + WebM
워크 분배 Stride modulo (stateless)
지원 포맷 BVH, FBX, GLB
