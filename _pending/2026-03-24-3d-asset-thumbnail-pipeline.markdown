## 문제 정의

사내 3D 에셋 라이브러리에는 6만 건 이상의 모션캡처 데이터가 있다. FBX, GLB, BVH 등 다양한 포맷의 3D 애니메이션 파일인데, 문제는 이걸 사용자에게 보여줄 방법이 없다는 것이었다. 3D 파일은 뷰어 없이는 내용을 알 수 없고, 6만 건을 일일이 열어볼 수도 없다.

에셋 라이브러리에 썸네일과 미리보기 영상이 필요했다. 정적 이미지(WebP) 하나와 애니메이션 미리보기(WebM) 하나. 6만 건 전부에 대해.

## 기술 선택: 왜 Puppeteer + Three.js인가

썸네일 생성 방식은 크게 두 가지가 있었다.

**서버사이드 렌더링(Blender headless 등)**을 쓰면 렌더링 품질은 좋지만, BVH/FBX/GLB 세 포맷을 모두 지원하려면 포맷별 변환 파이프라인이 필요하다. 유지보수 비용이 높고, 이미 프론트엔드에 Three.js 기반 뷰어가 있었기 때문에 로직을 재활용할 수 없다.

**Puppeteer + Three.js** 조합을 선택했다. 이유는 세 가지다.

첫째, 이미 서비스에서 쓰고 있는 Three.js 뷰어 코드를 거의 그대로 재활용할 수 있다. FBXLoader, GLTFLoader, 커스텀 BVHLoader2가 이미 동작하고 있었다.

둘째, Puppeteer로 헤드리스 브라우저를 띄우면 Canvas에서 프레임을 캡처해 FFmpeg에 파이프로 바로 넘길 수 있다. 별도의 중간 저장 없이 실시간 인코딩이 가능하다.

셋째, GPU 가속이 된다. `--enable-gpu`, `--use-gl=angle` 플래그로 WebGL 하드웨어 가속을 쓸 수 있고, GPU 없는 서버 환경에서는 SwiftShader로 폴백한다.

## 파이프라인 구조

전체 파이프라인은 3단계다.

**1단계: S3에서 에셋 다운로드.** 6만 건의 메타데이터(MongoDB에서 내보낸 JSON)를 읽고, S3에서 3D 파일을 로컬로 내려받는다. 동시 20개 다운로드, 실패 시 3회 재시도.

**2단계: 배치 캡처.** Vite 개발 서버로 React + Three.js 캡처 앱을 띄우고, Puppeteer가 이 앱에 접속해서 에셋을 하나씩 로드한다. Three.js가 애니메이션을 렌더링하면 Canvas에서 프레임을 WebP로 추출하고, FFmpeg stdin 파이프로 스트리밍해서 VP9 WebM으로 인코딩한다.

**3단계: S3에 업로드.** 생성된 WebP 썸네일과 WebM 영상을 다시 S3에 올린다. 이미 존재하는 파일은 스킵, 동시 20개 업로드, 실패 시 4회 재시도.

## 병렬 처리 설계

6만 건을 한 프로세스로 처리하면 시간이 너무 오래 걸린다. 멀티 프로세스 구조를 만들었다.

`start-batch.js`가 N개의 워커를 띄운다(기본 4개). 각 워커는 독립된 Puppeteer 브라우저 인스턴스를 가지고, 에셋을 stride 방식으로 분배받는다.

```
워커 0: 에셋 0, 4, 8, 12, ...
워커 1: 에셋 1, 5, 9, 13, ...
워커 2: 에셋 2, 6, 10, 14, ...
워커 3: 에셋 3, 7, 11, 15, ...
```

각 워커의 배치 사이즈(limit)는 200으로 제한했다. 전체 6만 건을 한번에 돌리면 로그 추적이 어렵고, 실패 시 재시작 범위가 너무 넓어진다. offset과 limit으로 구간을 나눠서 실행하면 실패한 구간만 재처리할 수 있다.

## 메모리 관리: 가장 어려웠던 부분

6만 건을 연속으로 처리하면 메모리가 계속 쌓인다. Three.js의 geometry, material, texture, skeleton이 GC되지 않고 누적된다. Puppeteer 프로세스가 수백 개의 에셋을 처리하다 보면 OOM으로 죽는다.

두 가지로 해결했다.

**페이지 재활용(Page Recycling).** 20개 에셋을 처리할 때마다 브라우저 페이지를 닫고 새로 연다. `renderer.forceContextLoss()`로 WebGL 컨텍스트를 강제 해제하고, Puppeteer 페이지를 새로 생성한다. 프로세스 자체를 재시작하는 것보다 빠르면서도 메모리를 효과적으로 회수한다.

**리소스 해제 정책 분리.** 모든 에셋에 동일한 기본 캐릭터 모델(Manny body)을 입힌다. 이 모델의 geometry와 material은 재사용하므로 해제하면 안 된다. 반면 각 에셋의 skeleton은 매번 새로 생성되므로 해제해야 한다. 'full'과 'skeletonOnly' 두 가지 dispose 정책을 만들어서 모델 유형에 따라 다르게 적용했다.

## 포맷별 처리의 까다로움

세 가지 3D 포맷을 하나의 파이프라인에서 처리해야 했다.

**FBX:** 스케일이 100배 큰 경우가 많아 0.01로 정규화. 좌표축이 Z-up인 파일은 Y-up으로 회전.

**GLB:** 가장 깔끔한 포맷. 이미 베이크된 애니메이션이 많아서 워밍업 프레임을 2개만 줘도 충분했다(다른 포맷은 15개).

**BVH:** 가장 까다로웠다. Three.js 기본 BVHLoader로는 부족해서 커스텀 BVHLoader2를 만들었다. 쿼터니언 변환에서 골반 회전 보정(X축 +π/2)이 필요했고, XYZ 회전을 Three.js 좌표계에 맞게 리매핑해야 했다.

포맷별로 워밍업 프레임 수를 다르게 설정한 것도 중요했다. 모션캡처 데이터(BVH, FBX)는 초기 15프레임 동안 포즈가 안정화되지 않는다. 이 구간을 캡처하면 부자연스러운 썸네일이 나온다. 반면 GLB는 2프레임이면 충분하다. 이걸 하나의 값으로 통일했다가 품질 문제가 생겨서 포맷별로 분리했다.

## FFmpeg 스트리밍과 백프레셔

캡처된 프레임을 파일로 저장했다가 FFmpeg에 넘기면 디스크 I/O가 병목이 된다. 대신 Puppeteer에서 캡처한 WebP 프레임을 FFmpeg의 stdin 파이프에 바로 스트리밍했다.

여기서 백프레셔 처리가 중요하다. FFmpeg의 인코딩 속도보다 프레임 생성이 빠르면 stdin 버퍼가 가득 차서 프레임이 유실된다. Node.js의 `stream.write()`가 `false`를 반환하면 `drain` 이벤트를 기다린 뒤 다음 프레임을 보낸다.

인코딩 설정은 VP9 코덱, CRF 18, 8스레드로 잡았다. CRF를 낮추면 품질은 올라가지만 인코딩 시간이 늘어나고, 어차피 썸네일 용도이므로 18이면 충분했다.

## 돌아보며

가장 큰 러닝 포인트는 "브라우저를 렌더 팜으로 쓸 수 있다"는 것이다. Puppeteer + Three.js 조합은 서버사이드 3D 렌더링의 진입 장벽을 크게 낮춘다. 이미 프론트엔드에 Three.js 뷰어가 있다면 그 코드를 거의 그대로 배치 처리에 활용할 수 있다.

다만 장기 실행 배치에서의 메모리 관리는 별도 설계가 필요하다. Three.js는 리소스를 자동으로 해제하지 않으므로, dispose 정책을 명시적으로 설계하지 않으면 수백 건 처리 후 반드시 OOM이 발생한다. 이건 프론트엔드 개발자가 간과하기 쉬운 부분이다.
